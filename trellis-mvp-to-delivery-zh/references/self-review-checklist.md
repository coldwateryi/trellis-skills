# 自我评审检查清单（MVP 到交付阶段）

本检查清单用于评估 MVP 差距审计输出是否满足小参数模型（如 qwen3.6 35b）的执行要求。

## 使用说明

1. 在完成每轮差距审计后，对照本清单逐项检查
2. 所有检查项必须通过（或明确标记"不适用"）才能进入用户确认阶段
3. 如有不通过项，标记具体问题并进行针对性改进

---

## 0. Trellis 0.6 Beta 工作流适配检查

### 0.1 工作流发现
- [ ] 如存在 `.trellis/workflow.md`，已读取
- [ ] 如存在 `.trellis/config.yaml`、`.trellis/.version`、`.trellis/.developer`，已检查
- [ ] 如现有任务包含 `design.md`、`implement.md`、`implement.jsonl`、`check.jsonl`，已识别并在补缺任务中保留延续
- [ ] Developer identity 设置以 `trellis init -u <name>` 为主要提示，`init_developer.py` 只作为旧版兜底

### 0.2 Spec 与任务产物新鲜度
- [ ] 列出了相关 `.trellis/spec/` 文件
- [ ] 列出了作为证据使用的已有任务产物
- [ ] Spec/任务产物新鲜度标记为 fresh/stale/missing/unknown
- [ ] Spec/任务产物过期或缺失时，已形成刷新任务或阻塞说明

### 0.3 规划产物门槛
- [ ] 每个补缺任务都说明必要产物（`prd.md`、`design.md`、`implement.md`、`implement.jsonl`、`check.jsonl`）
- [ ] 中/高复杂度任务已包含设计和实现产物，或已进一步拆小
- [ ] 需要在实现/检查前预加载稳定 specs、调研说明或外部上下文时，要求 JSONL 上下文清单

## A. 需求追踪矩阵质量检查

### A1. 状态判定准确性
- [ ] 每个需求都有明确状态（DONE/PARTIAL/MISSING/UNTESTED/UNCLEAR）
- [ ] 标记为 DONE 的需求有实现证据和测试证据
- [ ] 标记为 UNTESTED 的需求有实现证据和明确测试缺口
- [ ] 标记为 PARTIAL 的需求说明了哪些已完成、哪些缺失
- [ ] 标记为 MISSING 的需求确实没有实现证据
- [ ] 标记为 UNCLEAR 的需求给出了阻塞问题或歧义说明

### A2. 证据完整性
- [ ] 相关代码列出了具体文件路径（不是"相关实现"）
- [ ] 现有测试列出了具体测试类和方法名
- [ ] 缺口描述具体（不是"部分未实现"）
- [ ] 每个 DONE 的需求都有对应的代码和测试引用

### A3. 差距识别完整性
- [ ] 功能实现的差距已识别
- [ ] 测试覆盖的差距已识别
- [ ] 边界条件处理的差距已识别
- [ ] 错误处理的差距已识别

---

## B. MVP 完成度评估检查

### B1. 统计准确性
- [ ] 已完成需求数量准确
- [ ] 部分完成需求数量准确
- [ ] 未实现需求数量准确
- [ ] 已实现但未测试需求数量准确
- [ ] 不明确需求数量准确
- [ ] 百分比计算正确

### B2. 完成度判断
- [ ] MVP 完成度判断基于证据，不是主观估计
- [ ] 关键功能缺失已明确标识
- [ ] 测试覆盖不足已明确标识

---

## C. 补缺任务拆分质量检查

### C1. 拆分原则
- [ ] 一个任务只关闭一组高度相关的缺口
- [ ] 基础契约和 UI 展示分离（除非不可分离）
- [ ] 功能实现和最终验证分离
- [ ] 每个功能任务包含自己的基础测试
- [ ] 没有按文件拆分的任务

### C2. 复杂度评估
- [ ] 每个任务都标注了复杂度（低/中/高）
- [ ] 复杂度评估基于小模型能力
- [ ] 高复杂度任务已拆分或有详细步骤
- [ ] 低复杂度：有现成范例可照抄
- [ ] 中复杂度：有明确实现步骤
- [ ] 高复杂度：已拆分或每步定死

### C3. 依赖关系
- [ ] 每个任务的依赖关系已明确列出
- [ ] 没有循环依赖
- [ ] 测试任务依赖其测试的功能任务
- [ ] 最终验证任务依赖所有功能任务

### C4. 优先级
- [ ] 每个任务都分配了优先级（P0/P1/P2/P3）
- [ ] 阻塞性缺口优先级高
- [ ] 测试补缺任务优先级合理

---

## D. 交付任务 PRD 质量检查（关键！）

### D1. 占位符消除
- [ ] PRD 中没有任何 `<...>` 形式的占位符
- [ ] PRD 中没有"具体路径"、"相关文件"等抽象表述
- [ ] PRD 中没有"根据实际情况"、"视需要而定"等推理留白

### D2. Current Gap（当前缺口）描述
- [ ] 当前状态明确（DONE/PARTIAL/MISSING/UNTESTED/UNCLEAR）
- [ ] 证据部分列出了具体代码路径和测试路径
- [ ] 缺口部分具体说明缺少什么（不是"功能不完整"）
- [ ] 风险部分说明了为什么重要

### D3. Reference Implementation（参考实现）
- [ ] 如果 MVP 中有类似实现，指向了具体文件路径
- [ ] 如果 MVP 中无类似实现，明确写"无，按 Technical Notes 从零实现"
- [ ] 替换说明具体（如"复用 MVP 中的 UserService，扩展 authenticate 方法"）
- [ ] 不是"参考 MVP 相关代码"等模糊指引

### D4. File Manifest（文件清单）
- [ ] 列出了所有要操作的文件，精确到完整路径
- [ ] 每个文件标注了操作类型（新建/修改）
- [ ] 修改操作说明了具体位置（如"在 authenticate() 方法中添加授权检查"）
- [ ] 如涉及数据结构变更，附带了完整的字段表

### D5. Implementation Steps（实现步骤）
- [ ] 步骤是有序的（1, 2, 3...）
- [ ] 每步是具体动作，不是抽象目标
- [ ] 每步可独立验证
- [ ] Bug 修复的分支选择已定死（不是"根据bug情况修复"）
- [ ] 需要推理的决策已在步骤中定死

### D6. Implementation Scope（行为约束）
- [ ] 行为约束写成可判定的断言
- [ ] 边界条件的确切行为已明确
- [ ] 错误处理的确切行为已明确
- [ ] 兼容性要求已明确（不破坏已有 MVP 行为）

### D7. Acceptance Criteria（验收标准）
- [ ] 验收标准是可判定的断言，不是主观描述
- [ ] 包含需求 ID 达到预期状态的验收
- [ ] 包含正常路径的验收
- [ ] 包含异常路径的验收
- [ ] 包含边界条件的验收
- [ ] 包含不破坏 MVP 行为的验收

### D8. Self-Check Commands（自检命令）
- [ ] 提供了可直接运行的验证命令
- [ ] 命令是具体的，不是"运行测试"
- [ ] 命令附带了期望结果说明
- [ ] 命令无需人工判断即可确认通过/失败

### D9. Automated Tests Required
- [ ] 明确列出了需要的测试类型
- [ ] 每个测试点具体（被测方法 + 输入 + 期望输出）
- [ ] Regression Tests 覆盖了 MVP 已有行为
- [ ] 不是"添加必要测试"等抽象要求

## E. 小模型执行友好性检查

### E1. 决策点已定死
- [ ] 所有"用哪个注解"的决策已明确
- [ ] 所有"走哪条分支"的决策已明确（包括 bug 修复分支）
- [ ] 所有"命名规则"已明确
- [ ] 所有"表结构变更"已明确

### E2. MVP 复用指引明确
- [ ] 哪些 MVP 代码需要复用已明确
- [ ] 哪些 MVP 代码需要扩展已明确（扩展什么）
- [ ] 哪些 MVP 代码需要修改已明确（修改哪里、怎么改）
- [ ] 不破坏 MVP 行为的约束已明确

### E3. Forbidden（禁止事项）清单
- [ ] 明确列出了不该做的事
- [ ] 明确了不能破坏的 MVP 行为
- [ ] 明确了文件修改范围
- [ ] 明确了依赖约束

### E4. 技术栈和工具约束
- [ ] 使用的框架/库已在 Technical Notes 中列出
- [ ] 构建命令已明确
- [ ] 测试命令已明确（包括回归测试）

### E5. 上下文与设计左移
- [ ] 上下文清单列出了执行模型编辑前必须读取的 MVP 代码、测试和 specs
- [ ] 决策表定死了 bug 分支、命名、schema、API、校验等选择
- [ ] MVP 兼容性契约列出了必须保留的行为和回归检查
- [ ] 契约快照在编码前定义了 API/interface/data/state 行为
- [ ] 使用 `implement.jsonl` 时，条目列出稳定实现上下文文件，而不是源代码文件或步骤动作
- [ ] 使用 `check.jsonl` 时，条目列出稳定验证上下文文件，而不是测试命令

---

## F. 测试覆盖规划检查

### F1. 测试映射完整性
- [ ] 每个需求（REQ-xxx）都映射到至少一个测试
- [ ] 每个验收标准（AC-xxx）都映射到至少一个测试
- [ ] 测试类型分布合理（unit/integration/e2e/smoke/regression）

### F2. 回归测试要求
- [ ] MVP 已有行为有回归测试覆盖
- [ ] 回归测试具体（测试什么行为、期望什么结果）
- [ ] 不是"确保不破坏现有功能"等抽象要求

---

## G. Bug 分类和处理检查

### G1. Bug 分类准确性
- [ ] 每个发现的 bug 都有明确分类
- [ ] 分类依据是否阻塞当前需求验收
- [ ] 阻塞性 bug 计划在当前任务修复
- [ ] 非阻塞性 bug 记录风险或创建独立任务

### G2. Bug 修复规划
- [ ] 阻塞性 bug 的修复步骤已明确
- [ ] Bug 修复的分支逻辑已定死
- [ ] Bug 修复的验证方法已明确
- [ ] 非阻塞性 bug 的风险已记录

---

## H. 风险点检查

### H1. 风险识别
- [ ] 高风险模块已标记风险说明
- [ ] MVP 改动的兼容性风险已识别
- [ ] 测试覆盖不足的风险已识别
- [ ] 技术债务已记录到 Out of Scope

### H2. Out of Scope 清晰性
- [ ] Out of Scope 列出了明确排除的功能
- [ ] Out of Scope 列出了无法在规划阶段定死的点
- [ ] Out of Scope 不是"其他功能"等模糊表述
- [ ] 延期的非阻塞性 bug 已记录到 Out of Scope

---

## 检查结果判定

### 通过标准
- 所有适用的检查项都勾选 ✅
- 不适用的检查项已标记"N/A"并说明原因

### 不通过处理
- 未通过的检查项 → 标记为问题
- 生成问题清单（位置、问题描述、影响、改进建议）
- 进行针对性改进，进入下一轮评审

### 收敛条件
- 所有检查项通过 → 进入用户确认阶段
- 连续2轮无新问题 → 自动通过
- 超过5轮仍有问题 → 提示用户选择（换强模型/人工介入/接受风险）
