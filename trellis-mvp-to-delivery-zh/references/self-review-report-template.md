**位置**: REQ-003 的需求追踪矩阵条目

**改进建议**:
- 相关代码改为：`src/main/java/com/example/service/UserService.java (authenticate方法)`
- 现有测试改为：`src/test/java/com/example/service/UserServiceTest.java (testAuthenticate方法)`

---

### 问题 2：[类别: C2 复杂度评估] [严重程度: 中]

**位置**: 补缺任务 T-GAP-3（支付回调幂等性）

**问题描述**:
- 该任务标记为"中复杂度"
- 但涉及分布式事务、幂等性保证、并发控制，实际是高复杂度

**影响**:
- 小模型可能无法正确处理复杂逻辑
- 执行阶段容易出错

**改进建议**:
- 将任务拆分为：
  - T-GAP-3-1: 幂等性 token 生成和存储（低复杂度）
  - T-GAP-3-2: 支付回调幂等性检查（中复杂度，PRD 定死每个分支）
  - T-GAP-3-3: 幂等性测试（低复杂度）
- 或在 PRD 中把每个决策点定死

---

### 问题 3：[类别: D1 占位符消除] [严重程度: 高]

**位置**: REQ-005 补缺任务 PRD，Implementation Steps 第3步

**问题描述**:
- 原文："在 `<Service类>` 中添加 `<验证逻辑>`"
- 占位符未被替换

**影响**:
- 小模型不知道要修改哪个文件
- 小模型不知道要添加什么逻辑

**改进建议**:
- 替换为："在 `src/main/java/com/example/service/OrderService.java` 的 `createOrder()` 方法开头添加库存校验逻辑：if (inventory < quantity) throw new InsufficientStockException()"

---

### 问题 4：[类别: D3 Reference Implementation] [严重程度: 中]

**位置**: REQ-007 补缺任务 PRD，Reference Implementation

**问题描述**:
- 原文："参考 MVP 中类似的实现"
- 没有指向具体文件

**影响**:
- 小模型不知道参考哪个文件

**改进建议**:
- 替换为："复用 MVP 中的 `src/main/java/com/example/service/UserService.java` 的 `authenticate()` 方法模式，扩展为 `authorizeOrder()` 方法，增加订单权限检查"

---

### 问题 5：[类别: D5 Implementation Steps] [严重程度: 高]

**位置**: REQ-006 补缺任务 PRD，Implementation Steps 第2步

**问题描述**:
- 原文："修复订单状态不一致的 bug"
- 这是抽象目标，没有说明怎么修复、走哪条分支

**影响**:
- 小模型不知道 bug 根因在哪里
- 小模型不知道修复方案是什么

**改进建议**:
- 拆分为具体步骤：
  1. Bug 根因：支付成功回调时，订单状态更新失败但未回滚
  2. 修复方案：在 `PaymentCallbackHandler.java` 的 `handleSuccess()` 方法中：
     - 在更新订单状态前开启事务：`@Transactional`
     - 捕获更新失败异常，回滚支付状态
  3. 验证：模拟状态更新失败，确认支付状态也回滚

### 问题 6：[类别: E1 决策点已定死] [严重程度: 中]

**位置**: REQ-008 补缺任务 PRD，Implementation Steps

**问题描述**:
- 原文："根据 bug 情况选择修复策略"
- 留给小模型自由决策

**影响**:
- 小模型可能选择不一致的修复策略
- 可能引入新问题

**改进建议**:
- 明确决策："Bug 类型是空指针异常，修复策略是：在访问 user.getProfile() 前添加空值检查：if (user.getProfile() == null) throw new ProfileNotFoundException()"
