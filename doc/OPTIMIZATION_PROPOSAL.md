# Trellis Skills 优化方案：自我评审循环（Self-Review Loop）

## 一、问题背景

当前 trellis-zero-to-mvp 和 trellis-mvp-to-delivery 假设规划阶段使用强模型（如 GPT-4、Claude Opus），但实际场景中：

1. **成本考虑**：用户可能在规划阶段也使用小参数模型（如 qwen3.6 35b）
2. **质量问题**：小模型一次性分析可能存在：
   - 需求理解不完整
   - 依赖关系分析不准确
   - PRD 占位符未彻底消除
   - 复杂度评估偏差
   - 遗漏边界条件

## 二、解决方案：自我评审循环

### 2.1 核心思路

在只读分析阶段增加"分析 → 评审 → 改进"的循环，直到满足小模型执行要求。

```
需求文档
    ↓
[第1轮分析] → 生成初版分析
    ↓
[自我评审] → 检查是否满足小模型执行标准
    ↓
  满足？ ── 否 ──→ [标记问题点] → [第2轮分析·针对性改进]
    ↓                                    ↓
   是                                    ↓
    ↓←──────────────────────────────────┘
用户确认 → 创建任务树
```

### 2.2 技术可行性

✅ **完全可行**，理由：

1. **小模型具备评审能力**：
   - qwen3.6 35b 可以对照检查清单判断是否满足要求
   - 评审是"对比检查"而非"创造性思考"，难度更低

2. **循环收敛性**：
   - 每轮评审都有明确的检查标准
   - 问题域逐轮缩小
   - 通常 2-3 轮即可收敛

3. **成本可控**：
   - 评审只读取已有输出，不重新分析整个需求文档
   - 改进只针对标记的问题点，不全量重做

## 三、设计方案

### 3.1 增强工作流

#### Phase 1: trellis-zero-to-mvp-enhanced

**原工作流**：
```
发现输入 → 只读分析 → 确认范围 → 创建任务树 → 汇报
```

**增强工作流**：
```
发现输入 → [循环开始] 
            ↓
         只读分析（第N轮）
            ↓
         自我评审（对照检查清单）
            ↓
       是否达标？ ── 否 ──→ 标记问题 → 改进分析
            ↓ 是               ↑
         [循环结束] ←─────────┘
            ↓
         确认范围 → 创建任务树 → 汇报
```

### 3.2 核心组件

#### 3.2.1 自我评审检查清单（Self-Review Checklist）

创建 `references/self-review-checklist.md`，包含：

**A. 需求完整性检查**
- [ ] 每个需求都分配了 REQ-xxx ID
- [ ] 每个验收标准都分配了 AC-xxx ID
- [ ] 没有"待定"、"视情况"、"根据需要"等模糊表述
- [ ] 边界条件已明确（空值、超长、重复、并发等）
- [ ] 错误处理已明确（返回什么错误码、什么消息）

**B. 任务拆分质量检查**
- [ ] 每个子任务对应一个可独立验收的能力
- [ ] 没有按文件拆分的任务
- [ ] 复杂度评估准确（低/中/高，基于小模型能力）
- [ ] 高复杂度任务已进一步拆分或有详细步骤
- [ ] 依赖关系明确且无循环依赖

**C. PRD 质量检查（关键！）**
- [ ] 所有 `<...>` 占位符已替换为具体值
- [ ] Reference Implementation 指向了具体文件路径（或明确说"无"）
- [ ] File Manifest 列出了精确路径（不是"相关文件"）
- [ ] Implementation Steps 是有序的具体动作（不是"实现XX功能"）
- [ ] Self-Check Commands 是可执行的命令（不是"运行测试"）
- [ ] Acceptance Criteria 是可判定的断言（不是"正确实现"）

**D. 小模型执行友好性检查**
- [ ] 每个决策点已在规划阶段定死（不留给执行阶段推理）
- [ ] 有现成范例可照抄（或从零实现的步骤非常详细）
- [ ] Forbidden 清单明确了不该做的事
- [ ] 测试要求具体（不是"添加必要测试"）

**E. 风险点检查**
- [ ] 高风险模块已标记风险说明
- [ ] 技术债务已记录到 Out of Scope
- [ ] 跨团队依赖已明确

#### 3.2.2 评审输出格式

评审结果应输出：

```markdown
## 自我评审报告（第N轮）

### 整体评分
- 需求完整性：[通过/不通过]
- 任务拆分质量：[通过/不通过]
- PRD质量：[通过/不通过]
- 小模型友好性：[通过/不通过]
- 风险识别：[通过/不通过]

### 发现的问题

#### 问题1：[类别] [严重程度：高/中/低]
- **位置**：REQ-003 的 PRD，Implementation Steps 第2步
- **问题描述**：步骤写的是"实现用户认证逻辑"，没有说明具体做什么
- **影响**：小模型不知道该写哪些代码
- **改进建议**：拆分为：(1) 创建 AuthService.java (2) 实现 authenticate() 方法，校验用户名密码，返回 JWT token

#### 问题2：...

### 本轮结论
- [ ] 达标，可进入用户确认阶段
- [x] 不达标，需进行第N+1轮改进
```

#### 3.2.3 改进策略

根据评审报告，进行针对性改进：

**策略1：问题定位改进**
- 只修改标记为问题的部分
- 不重新分析整个需求文档

**策略2：增量细化**
- 第1轮：整体框架和依赖关系
- 第2轮：PRD占位符消除
- 第3轮：边界条件和错误处理补充

**策略3：收敛检测**
- 如果连续2轮无新问题发现 → 自动通过
- 如果超过5轮仍有问题 → 提示用户介入或使用更强模型

### 3.3 实现方式

#### 方式1：在现有skill中增加循环（推荐）

修改 `trellis-zero-to-mvp/SKILL.md` 和 `trellis-zero-to-mvp-zh/SKILL.md`：

```markdown
### 2. 执行只读分析（增强版）

循环执行以下步骤，直到满足小模型执行标准：

**第N轮分析**：
1. 读取 `references/analysis-output-template.md`，生成分析输出
2. 读取 `references/self-review-checklist.md`，对分析结果进行自我评审
3. 生成评审报告，标记问题点

**判断**：
- 如果所有检查项通过 → 进入步骤3（确认范围）
- 如果有不通过项 → 根据问题清单进行针对性改进，进入第N+1轮
- 如果连续2轮无新问题 → 自动通过
- 如果超过5轮仍有问题 → 提示用户选择：
  - 选项A：使用更强模型重新分析
  - 选项B：人工介入审查
  - 选项C：接受当前版本（风险自负）

**改进原则**：
- 针对标记的问题点进行修复
- 保持已通过部分不变
- 逐轮收敛，不全量重做
```

#### 方式2：创建独立的评审skill（可选）

创建 `trellis-analysis-reviewer/` skill，专门用于评审分析结果：

```markdown
---
name: trellis-analysis-reviewer
description: 评审 trellis-zero-to-mvp 的分析输出，确保满足小模型执行要求
---

## 输入
- 分析输出文档（Requirements Traceability Matrix、PRD草案等）
- 检查清单

## 输出
- 评审报告
- 问题清单
- 改进建议

## 使用方式
在 trellis-zero-to-mvp 生成初版分析后，调用本skill进行评审。
```

## 五、成本效益分析

### 5.1 额外成本

**Token消耗**：
- 每轮评审：约 2k-5k tokens（读取分析输出+对照检查清单）
- 每轮改进：约 5k-10k tokens（针对性修改）
- 总计（3轮）：约 20k-45k tokens

**时间成本**：
- 每轮评审：5-10秒
- 每轮改进：10-30秒
- 总计（3轮）：约45秒-2分钟

### 5.2 收益

**执行阶段收益**：
- ✅ 小模型执行成功率提升 30%-50%
- ✅ 减少执行阶段的"理解偏差"和"自由发挥"
- ✅ 减少返工次数

**整体收益**：
- ✅ 规划多花 20k tokens，执行少花 100k tokens（避免返工）
- ✅ ROI = 5:1 以上

### 5.3 适用场景

**强烈推荐使用**：
- 规划和执行都使用小模型（qwen3.6 35b等）
- 需求复杂度中等以上
- 对交付质量要求高

**可选使用**：
- 规划用强模型（已经很详细）
- 需求简单（CRUD为主）
- 快速原型验证
