# 需求落地能力增强设计建议

> 目标：在 **Trellis + trellis-skills** 基座上，把 **Superpowers 的工程纪律**（TDD/调试/评审）与 **CodeStable 的设计匠艺**（编排-计算分离/挂载点/结构健康度/架构档案）提炼为对 trellis-skills 的增强，**重点强化"任务生成后的需求落地能力"**，并**面向 qwen3.6 35b 这类小参数模型**。
>
> 背景与五框架对比见 [`FRAMEWORK_COMPARISON_REPORT.md`](./FRAMEWORK_COMPARISON_REPORT.md)。

---

## 0. 一句话方案

现状是**强模型在规划期把判断写进产物 → 小模型在执行期机械照搬**。trellis-skills 已是一个成熟的"规划期"系统（PRD 含决策表/参考实现/自检命令/禁止事项/自动化测试要求，design.md 含契约/非目标，implement.md 含文件计划/修改边界/失败恢复，外加自评收敛循环）。**它缺的恰恰是"执行期/验证期的纪律"和几个"设计透镜"**。

增强分三动作：

1. **给规划期产物注入 CodeStable 设计透镜**（编排-计算分离/挂载点/结构健康度/架构档案）——改 reference 模板。
2. **新增 3 个执行期技能**，把 Superpowers 的 TDD/调试/评审做成小模型走得动的刚性循环。
3. **用"角色分层模型分配"**（强模型规划+评审、小模型执行）把这一切落到 Trellis 的 per-agent `model` 配置上。

---

## 1. 核心洞察：为什么这么设计对小模型有效

> **小模型不擅长"判断"，擅长"照着窄路径执行有客观信号的机械步骤"。**

所以每一项增强都必须把"需要判断的活"转成"有红/绿信号的机械步"。TDD 的失败测试就是这个原则的最佳载体——它把不可验证的"实现这个需求"变成可验证的"让这条断言变绿"。

### 落地总原则（贯穿所有增强）

| 原则 | 含义 | 对应机制 |
| --- | --- | --- |
| **判断左移** | 所有需推理的决策（命名/分支/schema/落点/重构方案）在规划期由强模型定死 | 复用现有决策表/契约 + 新增编排分层/挂载点/重构方案 |
| **客观信号驱动** | 每步"完成"由机器信号判定（测试红绿/编译/退出码），不让小模型自评"做对没" | 新增 TDD 循环（red→green） |
| **窄路径 + 硬边界** | 把小模型锁在文件清单/禁止事项内；调试"一次一处、改完必重跑" | 复用修改边界 + 新增调试刚性脚本 |
| **角色分层模型分配** | 规划/评审用强模型，实现用小模型 | 落到 Trellis `agent.model` + `config` |
| **上下文最小化** | 小窗口只装"编辑前必读"的稳定上下文；一次只面对一个 AC | 复用 jsonl + TDD 单 AC 切片 |
| **失败可恢复+收敛兜底** | 调试有退出条件、评审有 critical 阻断、整体有"超 N 轮换强模型/人工" | 复用现有自评收敛机制 |

## 2. 集成点：任务生成后的"需求落地"编排

`zero-to-mvp / mvp-to-delivery` 生成主子任务后，对**每个子任务**走下面这条增强后的落地链（**粗体**为新增/增强）：

```
按依赖排序选一个子任务
   │  规划产物已含: PRD + design.md(【+编排-计算分离 +挂载点】) + implement.md(【+结构健康度预检/微重构第0步】) + jsonl
   ▼
【trellis-implement-tdd-zh】  对每个 AC: 写失败测试→看红→最小代码→看绿→提交  (每步客观信号, 小模型只追"变绿")
   │            └─(任一步失败)─▶【trellis-debug-systematic-zh】 刚性4步调试: 复现→定位→单假设验证→最小修复+回归 (一次一处)
   ▼
PRD「自检命令」全绿
   ▼
【trellis-review-twostage-zh】 Stage1 规范符合(对PRD机械核对) → Stage2 代码质量(强模型/checklist)
   │            └─(critical)─▶ 回 implement-tdd 修复
   ▼
【挂载点清单逐项勾选】(集成验证, 防"写了实现没接线") + 回归(check.jsonl)
   ▼
标记完成  ──(交付收尾, mvp-to-delivery)──▶【架构档案回写 .trellis/spec/.../ARCHITECTURE.md】
```

**关键设计**：执行/调试/评审三个新技能**不写进 PRD 给小模型读**，而是由**编排 agent（主会话，通常强模型）调用**，去驱动小模型执行子代理——因为小模型读不动也守不住一大段流程说明，流程必须由外层强模型"夹着走"。

## 3. 新增 3 个执行期技能（提炼自 Superpowers）

### 3.1 `trellis-implement-tdd-zh` —— 红绿重构落地

| | |
| --- | --- |
| **提炼自** | Superpowers `test-driven-development`（RED-GREEN-REFACTOR） |
| **触发** | 子任务进入实现、且复杂度≥中，或 PRD「自动化测试要求」非空 |
| **动作** | 把 PRD 的「验收标准/自动化测试要求」逐条转成 TDD 循环，驱动小模型执行子代理 |

**小模型适配核心**——把现有「实现步骤（…编译通过）」升级成**单 AC 红绿循环**，小模型每轮只面对一个断言：

```
对 PRD 里每个 AC-xxx，按顺序：
  1. 照抄 PRD「参考实现/测试范例」写一个针对 AC-xxx 的失败测试
  2. 运行该测试 → 必须看到红（失败信号=客观证据；没红=测试没写对，停）
  3. 写"刚好让它变绿"的最小代码（落点已由文件清单+编排分层定死）
  4. 运行该测试 → 看到绿（客观信号）
  5. 跑 PRD 自检命令保证没弄坏别的 → 记录/stage（Trellis executor 禁 git commit）
  6. 下一个 AC
```

为什么这是**最高杠杆**：失败测试把"理解并实现需求"（要判断）替换成"让这条断言变绿"（机械、可验证）。小模型不再需要知道"什么算做完"——绿就是做完。

### 3.2 `trellis-debug-systematic-zh` —— 刚性调试脚本

| | |
| --- | --- |
| **提炼自** | Superpowers `systematic-debugging`（4 阶段根因）+ `verification-before-completion` |
| **触发** | TDD 循环里某测试该绿不绿、或 PRD 自检命令失败 |
| **动作** | 用一段**小模型不能偏离的编号脚本**接管调试 |

**小模型适配核心**——小模型一旦"自由发挥"调试就会到处乱改、越改越坏。所以给的是**铁律脚本 + 硬约束**：

```
1. 固定失败信号：把报错/失败断言原文贴出，不复现不动手
2. 定位：只用"读栈→二分注释→加一行日志"三招缩小范围，禁止猜
3. 单一假设：写一句"我认为根因是 X"，先验证假设真假，再谈修
4. 最小修复：只改一处 → 重跑那个失败命令 → 仍红则回滚此次改动重来
铁律：一次只改一处；改完必重跑原失败命令；禁止动文件清单/禁止事项之外的文件
收尾(防御性)：回答"这个 bug 还能从别处再发生吗"——能则在 check.jsonl 记一条回归点
退出：绿了 / 已回滚 / 超3轮升级给强模型
```

### 3.3 `trellis-review-twostage-zh` —— 双阶段评审门

| | |
| --- | --- |
| **提炼自** | Superpowers `requesting-code-review` / `subagent-driven-development`（双阶段：先规范符合、再代码质量） |
| **触发** | 子任务自检全绿、标记完成之前 |
| **动作** | 两段独立评审，critical 阻断 |

**小模型适配核心**——**小模型评审自己的代码不可靠**，这一步是全链路里最该**升级到强模型**的地方（契合 Trellis check agent 可配 `model`）：

- **Stage 1 规范符合**（小模型+checklist 即可，机械核对）：逐条对照 PRD「验收标准/决策表/禁止事项/文件清单」——每个 AC 有对应测试且绿？改了清单外文件？违反禁止事项？挂载点是否都接线？
- **Stage 2 代码质量**（**强模型**做）：对照 `.trellis/spec/guides/` 工程规范（这里吃到 §4 注入的 CodeStable 设计纪律）；按严重度输出 issue，critical 打回 3.1。

## 4. 增强现有技能/模板（注入 CodeStable 设计匠艺）

这些是**改 trellis-skills 的 reference 模板**（不是新技能），让设计透镜在规划期就进入产物。涉及文件：`references/planning-artifacts-template.md`、`references/child-prd-template.md`、`references/self-review-checklist.md`，以及 mvp-to-delivery 的 `references/final-acceptance-template.md`。

### 4.1 `design.md` 增加两节：编排-计算分离 + 挂载点

- **编排-计算分离**：标注本次改动哪些属**编排层**（主流程/控制流/workflow），哪些属**计算层**（纯算法/纯函数），各自指向文件清单里的落点。
  - *小模型收益*：直接消除"代码该放哪"的判断——"先搭编排骨架，再逐个填计算节点"，落点已定死。
- **挂载点清单**（3–5 条，判据"删了它特性就消失"）：路由注册 / 配置项 / 事件订阅 / DI 绑定 / 菜单入口…
  - *小模型收益*：一份**显式接线 checklist**。小模型最常翻车的是"写了实现却没接线"，挂载点把接线变可勾选项（对应 §2 的"挂载点逐项勾选"）。

### 4.2 `implement.md` / 子任务 PRD 增加"结构健康度预检 + 微重构第 0 步"

- 规划期对每个要改的文件做**机械阈值预检**：目标文件 > N 行（默认 400）？目标目录 > M 个文件（默认 15）？命中则在 implement.md **第 0 步**放一个"只搬不改行为"的微重构（provable refactor，编译器全绿）。
- *小模型铁律*：重构方案（搬什么→搬到哪→怎么验证不变）**必须由强模型在规划期定死**，执行期小模型只做机械移动 + 跑编译。**绝不让小模型自己判断"要不要/怎么重构"。**

### 4.3 架构档案回写（CodeStable `architecture/` 等价物，落进 Trellis spec）

- design.md 涉及新模块/跨模块接口时，提示在 `.trellis/spec/<pkg>/ARCHITECTURE.md` 加指向；
- 交付收尾（增强 `mvp-to-delivery` 的 `final-acceptance`）时回写架构现状。
- 既拿到 CodeStable 的长效架构档案，又走 Trellis 的团队共享/版本化/spec 自动注入。

### 4.4 `self-review-checklist.md` 增加 F、G 两组（复用现有收敛循环）

- **F. TDD 就绪**：每个 AC 是否有"先写的失败测试"定义？测试命令可直接运行？
- **G. 落地纪律**：编排-计算分层是否标注落点？挂载点清单是否齐（可勾选）？结构健康度预检是否做？复杂任务是否挂上调试脚本/评审门？

## 5. 角色分层与模型分配（让小模型真正可用的关键）

Trellis 的 agent 定义里有 `model:` 字段，config 有 `context_window`——天然支持**按角色分配模型**：

| 角色 | 用什么模型 | 干什么 |
| --- | --- | --- |
| 规划（zero-to-mvp / mvp-to-delivery + §4 增强） | **强模型**（Opus/GPT-5.5 等） | 定死一切判断，产出窄路径产物 |
| 实现（implement-tdd） | **小模型 qwen3.6 35b** | 红绿循环机械照搬 |
| 调试（debug-systematic） | 小模型为主，超 3 轮**升强模型** | 刚性脚本定位修复 |
| 评审 Stage2（review-twostage） | **强模型** | 代码质量把关 |

一句话：**强模型把"判断"全部前置和兜底，小模型只在中间做"有红绿信号的机械执行"。**

---

## 6. 对多人团队协同的额外收益

- **TDD 测试 + 双阶段评审门** = 团队信任小模型/他人产出的客观依据，PR 可审；
- **挂载点清单** = 跨人交接时的集成验证单，减少"我以为接好了"；
- **架构档案回写到 spec** = 全队共享、版本化、自动注入的长效架构记忆；
- **角色分层模型分配** = 算力受限的成员（只有本地 qwen）也能按统一质量贡献代码。

## 7. 落地范围与顺序

**本轮实现范围（中文 -zh 技能优先）**：

1. **模板增强（§4.1–4.4）**：纯文档增强、零运行时改动、立刻让产物带上设计透镜——投入最小、收益即时。
2. **新增 `trellis-implement-tdd-zh`**：单点收益最高，只依赖 PRD 已有的「验收标准/自动化测试要求」。
3. **新增 `trellis-debug-systematic-zh` + `trellis-review-twostage-zh`**：补齐执行期闭环。
4. **更新 README**：技能概览表 + 推荐工作流补上落地链。

**后续（follow-up）**：

- 把以上增强**同步移植到英文技能** `trellis-zero-to-mvp` / `trellis-mvp-to-delivery` 及新增英文版执行期技能（保持 EN/ZH 对称）。
- 评估 gsd-core 式的"原子锁 + STATE.md"在多 agent 并行落地时的借鉴价值。

**实现纪律**：每个新技能按 trellis-skills 既有规范带 `references/` + 自评清单分组，复用收敛循环；改动后用本仓库自身的 self-review 方法论多轮收敛检查。
