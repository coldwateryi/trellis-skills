# Stage 2 · 代码质量与设计纪律核对清单

对照 `.trellis/spec/guides/` 与设计纪律评审代码本身。**应由强模型执行**。严重度：critical（阻断）/ major（应修）/ minor（记录后续）。

## 编排-计算分离（源自 CodeStable 设计纪律）

- [ ] 编排逻辑（主流程/控制流/分支编排）与计算逻辑（纯算法/纯函数/数据变换）没有混在同一处。
- [ ] 纯计算没有被塞进编排层导致无法单测；可独立测试的逻辑确实可独立测试。
- [ ] 新代码落在与 `design.md` 编排-计算分层一致的位置。

## 结构健康度（源自 CodeStable 设计纪律）

- [ ] 没有往本就偏胖（超阈值）的文件继续堆砌，使其更难分清职责。
- [ ] 没有制造"什么都装的筐"式万能文件/工具类。
- [ ] 目录归属合理，没有把新文件随手丢进无关目录。

## 简化与复用

- [ ] 没有重复实现已存在的能力（应复用参考实现/现有工具）。
- [ ] 没有未被需求要求的过度抽象 / 预留扩展点（YAGNI）。
- [ ] 没有明显可去除的重复代码（DRY），且去重不损害可读性。

## 正确性（不止 happy path）

- [ ] 边界条件（空/超长/重复/非法输入）按 `prd.md` 行为约束真正处理，而非只让正常路径测试变绿。⛔ 漏处理已声明边界为 critical
- [ ] 错误路径返回 `prd.md` 指定的确切错误码/消息。
- [ ] 并发/顺序假设（若适用）成立，无明显竞态。
- [ ] 未破坏既有行为（回归点见 `check.jsonl`）。⛔ 破坏既有行为为 critical

## 规范符合（spec/guides）

- [ ] 命名 / 分层 / 模块边界符合 `.trellis/spec/` 相关规范。
- [ ] 错误语义 / 日志 / 可观测点符合项目约定。
- [ ] 测试风格与既有测试一致（断言粒度、命名、组织）。

## 架构影响（若涉及新模块/跨模块接口）

- [ ] 新模块对外暴露/依赖关系清晰，与既有架构一致。
- [ ] 已提示在 `.trellis/spec/<pkg>/ARCHITECTURE.md` 加指向（交付收尾回写）。

> 评审者只产出问题清单，不直接改代码。critical 必阻断。
