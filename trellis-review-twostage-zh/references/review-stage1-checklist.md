# Stage 1 · 规范符合核对清单

机械核对"实现是否照 `prd.md` 规划做了"。可由小模型逐项执行。标 ⛔ 的不符为 **critical**（阻断）。

## 验收标准覆盖

- [ ] 每条 `AC-xxx` 都有对应测试。⛔ 缺失为 critical
- [ ] 所有 AC 对应测试当前为绿。⛔ 有红为 critical
- [ ] 没有用"删测试/弱化断言"的方式凑绿。⛔ 有则 critical

## 文件清单符合

- [ ] 改动的每个文件都在「文件清单」内。⛔ 改了清单外文件为 critical（除非已报告并获批）
- [ ] 「文件清单」中标"新建"的确为新建、标"修改"的改在指定位置。

## 禁止事项符合

- [ ] 未新建「禁止事项」点名的已有基类/工具类（应复用参考实现）。⛔ critical
- [ ] 未引入未在技术备注列出的新依赖/框架。⛔ critical
- [ ] 未触碰禁止事项点名的其他红线。

## 决策表符合

- [ ] 注解 / 命名 / schema / 分支等均按「决策表」选定方案，无自行改选。

## 挂载点接线（若 design.md 有挂载点清单）

- [ ] 挂载点逐项已接线：路由注册 / 配置项 / 事件订阅 / DI 绑定 / 菜单入口等。⛔ 漏接为 critical（功能等于没上线）

## 范围符合

- [ ] 未实现「范围外」点名的功能。
- [ ] 未越界实现其他 AC / 其他任务的内容。

> 任一 ⛔ 命中 → Stage 1 即判 critical，可不必进入 Stage 2，直接打回修复。
