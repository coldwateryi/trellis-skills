# TDD 红绿循环详规

逐条验收标准（AC）执行。本文件是给执行体（可能是 qwen3.6 35b 小模型）的窄路径脚本——照着做，不要发挥。

## 单条 AC 的循环

### 步骤 1 · RED（写一个失败的测试）

- 从 `prd.md`「参考实现/测试范例」找到可照抄的测试，复制它，按本 AC 改输入与断言。
- 测试只针对**这一条 AC** 的可观察结果（返回值 / 状态变化 / 错误码 / 落库结果），不贪多。
- 断言写成确定值，禁止"大概""非空即可"这类弱断言（除非 AC 本身就是非空校验）。
- 测试命令取自 `prd.md`「自检命令」或「自动化测试要求」，必须可直接运行。

### 步骤 2 · 看红（确认测试真的失败）

- 运行该测试，**必须看到失败**。
- **看不到失败 = 测试没覆盖到目标行为**：回到步骤 1 修测试，不要往下走。
- 失败信息要指向"功能未实现"（如断言不符 / 方法不存在），而不是"测试写错了"（如导入错误 / 语法错）。后者先修测试。

### 步骤 3 · GREEN（写刚好变绿的最小代码）

- 落点已定：新代码放哪由 `prd.md`「文件清单」+ `design.md`「编排-计算分离」决定。编排层改编排层，计算层改计算层，不混。
- 决策已定：命名 / 分支 / schema / 注解等照 `prd.md`「决策表」，不自行选择。
- 只写"让当前这条测试变绿所需的最小代码"，不顺手实现别的 AC、不加未要求的抽象。

### 步骤 4 · 看绿（确认通过）

- 再次运行该测试，看到通过。
- 仍红且不显然 → 触发 `trellis-debug-systematic-zh`，修复后回到本步骤重跑。

### 步骤 5 · 自检（确认无回归）

- 运行 `prd.md`「自检命令」全集 + 项目 lint / type-check。
- 任一项失败 = 本次改动弄坏了别处 → 触发 `trellis-debug-systematic-zh`。

### 步骤 6 · 收敛（删除不必要复杂度）

- 如果本 AC 引入新抽象、新依赖、新文件、重复实现或未来扩展点，读取 `references/minimal-implementation-pass.md`。
- 优先删除不需要存在的代码，优先标准库/平台/已有依赖，禁止为了“简单”删除输入校验、错误处理、安全、可访问性和已声明边界。
- 如果保留有已知上限的简化实现，用 `trellis-minimal:` 注释说明上限和升级条件。
- 改完后重跑本 AC 测试和相关自检命令。

### 步骤 7 · 记录（暂存，不提交）

- 进度表把 `AC-xxx` 标为 done。
- 暂存改动（如 `git add`），**不要 `git commit`**。
- 进入下一条 AC。

## 铁律

- 没有失败测试，不写实现代码。
- 一次只让一条 AC 变绿。
- 不改「文件清单」「禁止事项」之外的文件；需要时停下来报告。
- 不删既有测试来"凑绿"；不弱化断言来"凑绿"。
- 不为了最小化删除 `prd.md` 要求的安全、校验、错误处理、可访问性或回归保护。
- 不 commit / push / merge。

## 反模式（出现即停，回到正轨）

- **先写实现再补测试**：违反 RED 优先，删掉未测代码，从步骤 1 重来。
- **一口气改多文件让多条 AC 一起绿**：失去定位能力，拆回单 AC。
- **看不到红就继续**：测试没在测真东西，先修测试。
- **绿不了就改测试断言迁就实现**：除非确认是测试写错；否则是实现没满足 AC，应改实现或走调试。
- **顺手重构无关代码**：超出本 AC，记到任务备注，留给评审/后续，不在绿循环里做。
- **把最小实现当代码高尔夫**：压缩到难读、削掉检查或隐藏边界，违反 Trellis 质量门。
