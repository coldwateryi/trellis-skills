# 最小实现收敛门

在所有 AC 测试变绿后执行。本文件吸收 Ponytail 的“最小可行实现阶梯”，但按 Trellis 的 TDD 和需求追踪约束重写：目标是删除不必要复杂度，不是压缩正确性。

## 使用时机

- 单条 AC 变绿后，如果改动明显引入新抽象、新依赖或新文件，立即执行一次轻量检查。
- 所有 AC 全绿、运行全量自检前，必须执行一次完整检查。
- 评审前必须把检查结果写入 `<task-dir>/tdd-progress.md`。

## 最小实现阶梯

按顺序检查，命中上层就不要走下层：

1. **这段代码需要存在吗？** 如果只是“以后可能需要”，删除。
2. **标准库已经覆盖吗？** 优先用标准库能力，删除手写实现。
3. **平台/框架原生能力已经覆盖吗？** 优先用浏览器、数据库、框架、语言内建特性。
4. **项目已安装依赖已经覆盖吗？** 复用现有依赖；不要为了少量逻辑新增依赖。
5. **能否更直接表达？** 删除一实现一接口、单调用者工厂、没人设置的配置项、无第二实现的抽象层。
6. **只保留满足 AC 的最小代码。** 不顺手实现未来需求，不预留扩展点。

## 不能删除或弱化

以下内容不是复杂度，不能为了“简单”删除：

- `prd.md` 明确要求的行为。
- trust boundary 的输入校验。
- 防止数据丢失的错误处理。
- security / auth / permission 检查。
- accessibility 基础行为。
- 已声明的边界条件和错误路径。
- 已有行为的回归保护。
- 非平凡逻辑的最小可运行检查。
- 硬件、时钟、传感器或外部系统需要的校准/容差参数。

## 标记有意简化

如果保留了一个有上限但当前足够的简化实现，在代码旁写短注释：

```text
trellis-minimal: <当前简化点>；当 <明确触发条件> 时升级为 <升级路径>
```

只在简化有已知上限时写注释。显而易见的简单代码不需要注释。

## 输出到进度表

在 `<task-dir>/tdd-progress.md` 的“最小实现收敛”区写：

- 删除了什么。
- 复用了什么标准库/平台/已有依赖。
- 保留了哪些不能删除的安全/校验/错误处理。
- 是否存在 `trellis-minimal:` 注释及升级条件。

## 停止条件

如果最小实现检查发现当前 PRD 要求可能过度设计，不要自行改需求。记录建议并交给编排会话或用户确认。
