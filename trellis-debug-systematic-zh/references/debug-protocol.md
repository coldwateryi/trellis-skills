# 系统化调试脚本细则

给执行体（可能是 qwen3.6 35b 小模型）的窄路径调试脚本。照着做，禁止跳步、禁止猜。

## 三招定位法（按代价从低到高，命中即停）

| 招 | 怎么做 | 适用 |
| --- | --- | --- |
| 读栈 | 直接看报错堆栈/失败断言指向的文件与行 | 有明确堆栈/断言位置 |
| 二分注释 | 对可疑代码段二分注释或短路，看失败是否消失 | 范围大、无明确堆栈 |
| 加一行日志 | 在缩小后的范围打印关键变量实际值，对比期望 | 怀疑是数据/状态不对 |

> 三招都用完仍定位不到 → 升级，不要开始乱改。

## 单一假设的写法

- 必须具体到**变量 / 分支 / 契约**：`我认为根因是 user.role 在 X 分支为 null`，不是"权限逻辑有问题"。
- 先**验证真假**再修：看该变量实际值 / 查契约定义 / 跑一个更小探针。
- 一次只持有一个假设。证伪就记入"已排除"，换下一个。

## 最小修复的边界

- 只改已验证根因的**那一处**。
- 改完**立刻重跑原失败命令**——这是唯一的验证方式。
- 仍红 → **回滚本次改动**再换假设。绝不在一个无效改动上继续叠加改动。
- 不借调试之名重构、改签名、改无关文件。

## 防御性回归

修绿后必答一句："这个 bug 还能从别的入口再发生吗？"

- 能 → 在任务的 `check.jsonl` 追加一行回归点，并尽量补一个回归测试覆盖该入口。
- 不能 → 记录"已局部封闭"。

## 反模式（出现即停）

- **霰弹式修改**：一次改好几处看哪处生效——失去因果，回到一次一处。
- **不重跑就宣布修好**：没重跑原失败命令 = 没验证。
- **带着未验证的假设改代码**：先验证假设真假。
- **改测试断言迁就实现**：除非确认测试本身写错，否则是实现没满足 AC。
- **在错误改动上继续叠加**：先回滚。
- **超 3 轮还在试**：停止升级，附已排除项。
