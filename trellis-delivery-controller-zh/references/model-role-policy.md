# 模型角色策略

## 默认分工

| 阶段 | 推荐执行者 | 原因 |
| --- | --- | --- |
| 需求分析和任务拆分 | 强模型 | 需要判断边界、依赖、拆分和执行规格 |
| 自我评审 checklist | 小模型或中等模型 | 主要是机械核对 |
| TDD 实现 | 小模型 | 只追求测试红绿信号 |
| 系统调试前 3 轮 | 小模型 | 按固定脚本定位和最小修复 |
| 复杂调试接管 | 强模型 | 超过 3 轮说明根因可能需要判断 |
| Review Stage 1 | 小模型 | 对照 PRD 机械核对 |
| Review Stage 2 | 强模型或人工 | 需要判断设计质量、结构健康和正确性风险 |
| Final Acceptance | 强模型或人工 | 需要综合需求、证据和残余风险 |

## 小模型禁止事项

- 不自行扩大范围。
- 不自行改 File Manifest 外文件。
- 不跳过失败测试直接写实现。
- 不通过删除测试或弱化断言制造绿灯。
- 不自行完成 Stage 2 代码质量评审。
- 不把没有测试证据的需求标记为 `DONE`。

## 升级条件

- 调试超过 3 轮仍失败。
- 评审发现 critical。
- 需求或架构决策无法从文件证据定死。
- 同一 REQ 连续 2 轮无进展。
- 需要判断跨模块架构、数据一致性、安全、权限、支付或基础设施风险。
