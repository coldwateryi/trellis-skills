## 三、发现的问题

### 问题 1：[类别: A3 边界条件] [严重程度: 高]

**位置**: REQ-003（用户注册功能）

**问题描述**:
- PRD 中未说明用户名为空时的行为
- PRD 中未说明用户名超过50字符时的行为
- PRD 中未说明用户名重复时返回的具体错误消息

**影响**:
- 小模型可能随意处理边界条件，导致不一致的行为
- 无法验证边界条件是否正确实现

**改进建议**:
在 REQ-003 的 PRD 中补充：
- 用户名为空：返回 400，错误消息"用户名不能为空"
- 用户名超过50字符：返回 400，错误消息"用户名长度不能超过50字符"
- 用户名重复：返回 409，错误消息"用户名已存在"

---

### 问题 2：[类别: B2 复杂度评估] [严重程度: 中]

**位置**: Task T3（订单支付流程）

**问题描述**:
- 该任务标记为"中复杂度"，但涉及支付回调、事务一致性、幂等性保证
- 实际上对小模型来说是高复杂度任务

**影响**:
- 小模型可能无法正确处理复杂的事务逻辑
- 执行阶段容易出错

**改进建议**:
- 将 Task T3 拆分为：
  - T3-1: 订单支付请求（低复杂度）
  - T3-2: 支付回调处理（中复杂度，PRD 中定死每个分支）
  - T3-3: 支付状态查询（低复杂度）
- 或者在 PRD 中把每个决策分支都定死（如"收到成功回调走分支A，收到失败回调走分支B"）

### 问题 3：[类别: C1 占位符消除] [严重程度: 高]

**位置**: REQ-005 的子任务 PRD，Implementation Steps 第2步

**问题描述**:

- 占位符 `<Controller类>` 和 `<方法>` 未被替换

**影响**:
- 小模型不知道要修改哪个文件
- 小模型不知道要添加什么方法

**改进建议**:
- 替换为："在 `src/main/java/com/example/OrderController.java` 中添加 `createOrder(OrderDTO dto)` 方法"

### 问题 5：[类别: C3 File Manifest] [严重程度: 高]

**位置**: REQ-004 的子任务 PRD，File Manifest

**问题描述**:
- File Manifest 中写的是"修改 Service 层相关文件"
- 没有列出具体文件路径

**影响**:
- 小模型不知道要修改哪些文件

**改进建议**:
- 替换为具体路径：
  - 修改 `src/main/java/com/example/service/OrderService.java`
  - 修改 `src/main/java/com/example/service/impl/OrderServiceImpl.java`

---

### 问题 6：[类别: C4 Implementation Steps] [严重程度: 高]

**位置**: REQ-006 的子任务 PRD，Implementation Steps 第3步

**问题描述**:
- 原文："实现订单状态机逻辑"
- 这是抽象目标，不是具体动作

**影响**:
- 小模型不知道如何实现状态机
- 可能自由发挥，导致实现偏差

**改进建议**:
- 拆分为具体步骤：
  1. 在 OrderStatus 枚举中定义状态：PENDING, PAID, SHIPPED, COMPLETED, CANCELLED
  2. 在 Order 实体类中添加 status 字段（类型 OrderStatus）
  3. 在 OrderServiceImpl 中添加状态转换方法：
     - transitionToPaid(): 从 PENDING → PAID，否则抛异常
     - transitionToShipped(): 从 PAID → SHIPPED，否则抛异常
     - ...

### 问题 7：[类别: D1 决策点已定死] [严重程度: 中]

**位置**: REQ-008 的子任务 PRD，Implementation Steps
