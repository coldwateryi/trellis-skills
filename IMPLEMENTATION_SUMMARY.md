## 二、核心功能设计

### 2.1 工作流增强

**原工作流**：
```
需求文档 → 只读分析 → 用户确认 → 创建任务树
```

**增强工作流**：
```
需求文档 → [循环开始]
            ↓
         第N轮分析
            ↓
         自我评审（对照检查清单）
            ↓
       达标？ ── 否 ──→ 标记问题 → 针对性改进
         ↓ 是               ↑
      [循环结束] ←─────────┘
         ↓
      用户确认 → 创建任务树
```

### 2.2 收敛机制

- ✅ 所有检查项通过 → 进入用户确认
- ✅ 连续2轮无新问题 → 自动通过
- ❌ 超过5轮仍有问题 → 提示用户选择（换模型/人工介入/接受风险）

### 2.3 检查清单（5大类45项）

| 类别 | 检查项数量 | 关键点 |
|-----|-----------|--------|
| A. 需求完整性 | 约15项 | REQ/AC ID、边界条件、错误处理 |
| B. 任务拆分质量 | 约12项 | 按能力拆分、复杂度评估、依赖关系 |
| C. PRD 质量 | 约18项 | **核心**：消除占位符、具体路径、可执行步骤 |
| D. 小模型友好性 | 约8项 | 决策定死、范例可照抄、禁止清单 |
| E. 风险识别 | 约5项 | 高风险标记、Out of Scope 清晰 |

## 三、关键设计亮点

### 3.1 问题定位精确

评审报告要求问题定位到：
- 具体的 REQ-xxx 或 AC-xxx
- 具体的 Task ID
- 具体的 PRD 章节（如 Implementation Steps 第2步）

**示例**：
```
问题3：[类别: C1 占位符消除] [严重程度: 高]
位置：REQ-005 的子任务 PRD，Implementation Steps 第2步
问题描述：原文："在 `<Controller类>` 中添加 `<方法>`"
改进建议：替换为"在 `src/main/java/com/example/OrderController.java` 中添加 `createOrder(OrderDTO dto)` 方法"
```

### 3.2 针对性改进

- **不重新分析整个需求文档**（节省token）
- **只修改标记为问题的部分**
- **保持已通过部分不变**
- **逐轮收敛，不全量重做**

### 3.3 改进追踪

评审报告包含：
- 已在本轮修复的问题（来自上一轮）
- 本轮新发现的问题
- 形成闭环追踪

### 3.4 弱模型友好

检查清单本身设计为：
- ✅ 简单的判断题（有/无、是/否）
- ✅ 对照检查，不需要创造性思维
- ✅ 适合 qwen3.6 35b 等小模型执行

## 五、使用方式

### 5.1 对于 AI 助手（如 Claude）

调用 trellis-zero-to-mvp-zh 或 trellis-zero-to-mvp skill 时：

1. **生成分析**：按 analysis-output-template.md 生成初版分析
2. **自我评审**：读取 self-review-checklist.md，逐项检查
3. **生成报告**：按 self-review-report-template.md 格式输出评审报告
4. **判断达标**：
   - 如果达标 → 请求用户确认
   - 如果不达标 → 根据问题清单进行改进，重复步骤2-4
5. **用户确认后**：创建 Trellis 任务树

### 5.2 对于用户

无需额外操作，工作流自动进行：
- 系统会显示每轮的评审报告
- 用户只需在最终达标后确认一次
- 如果超过5轮，系统会提示选择

## 七、技术债务和已知限制

### 7.1 当前限制

1. **手动触发**：当前需要 AI 助手主动执行评审循环，未完全自动化
2. **检查清单覆盖面**：基于经验设计，可能遗漏某些边缘问题
3. **收敛判定**：固定阈值（5轮），可能需要根据任务复杂度动态调整

### 7.2 未来改进方向

1. **自动化程度**：开发独立脚本，自动执行检查清单中的部分项（如占位符检测）
2. **智能收敛**：根据问题严重程度和数量动态调整轮数上限
3. **问题模式匹配**：从历史评审中学习，预判常见问题
